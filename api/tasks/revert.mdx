## Endpoint

<CodeGroup>
```http
POST /api/v1/human-tasks/{task_id}/revert
```
</CodeGroup>

Reverts a human task, sending the workflow back to a previous step. This action is only available if the step definition has `allow_revert` enabled and a revert transition is configured.

## Authentication

This endpoint requires authentication. Include a valid Bearer token in the Authorization header.

## Path Parameters

<ParamField path="task_id" type="uuid" required>
  The unique identifier of the task to revert
</ParamField>

## Request Body

<ParamField body="actionType" type="string">
  Not required - automatically set to `revert`
</ParamField>

<ParamField body="remark" type="string">
  Optional comment explaining why the task is being reverted. **Required** if the step definition has `remark_required_on_revert` set to true.
</ParamField>

<ParamField body="payload" type="object" default="{}">
  Optional additional data to include with the revert action. Can capture revision requests or feedback.
</ParamField>

<ParamField body="actionCode" type="string">
  Not used for revert actions (only for custom actions)
</ParamField>

## Response Schema

<ResponseField name="workflowInstanceId" type="uuid" required>
  ID of the workflow instance
</ResponseField>

<ResponseField name="stepInstanceId" type="uuid" required>
  ID of the step instance that was reverted
</ResponseField>

<ResponseField name="actionType" type="string" required>
  The action that was taken: `revert`
</ResponseField>

<ResponseField name="workflowStatus" type="string" required>
  Current status of the workflow after the revert: typically `waiting` or `running`
</ResponseField>

<ResponseField name="nextStepCode" type="string">
  Code of the step the workflow reverted to
</ResponseField>

## Revert Availability

The `revert` action is only available if:

1. The step definition has `allow_revert` set to `true`
2. A transition with `action_type = 'revert'` exists from the current step
3. The task's `availableActions` array includes `"revert"`

<Note>
Revert actions are commonly used when an approver needs more information or clarification before making a final decision. The workflow sends the request back to an earlier step (often the submitter) for revision.
</Note>

## Revert vs. Reject

| Action | Purpose | Workflow Impact |
|--------|---------|----------------|
| **Revert** | Send back for revision/clarification | Workflow continues after revisions |
| **Reject** | Decline the request | May terminate workflow or follow rejection path |

## Cancellation of Other Tasks

When a task is reverted, all other `open`, `queued`, or `claimed` tasks in the same step instance are automatically cancelled.

## Error Handling

| Status Code | Description |
|-------------|-------------|
| 200 | Task reverted successfully |
| 403 | User is not assigned to this task |
| 404 | Task not found |
| 409 | Task status is not `open` (already completed, cancelled, or queued) |
| 422 | Remark is required but not provided, or revert action not allowed/configured |

<Note>
If `remark_required_on_revert` is enabled, the `remark` field is mandatory. This ensures the submitter understands what needs to be revised.
</Note>
