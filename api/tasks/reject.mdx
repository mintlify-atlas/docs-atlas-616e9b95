## Endpoint

<CodeGroup>
```http
POST /api/v1/human-tasks/{task_id}/reject
```
</CodeGroup>

Rejects a human task assigned to the current user. This convenience endpoint sets the action type to `reject` automatically and follows the rejection transition path defined in the workflow.

## Authentication

This endpoint requires authentication. Include a valid Bearer token in the Authorization header.

## Path Parameters

<ParamField path="task_id" type="uuid" required>
  The unique identifier of the task to reject
</ParamField>

## Request Body

<ParamField body="actionType" type="string">
  Not required - automatically set to `reject`
</ParamField>

<ParamField body="remark" type="string">
  Optional comment explaining the rejection. **Required** if the step definition has `remark_required_on_reject` set to true.
</ParamField>

<ParamField body="payload" type="object" default="{}">
  Optional additional data to include with the rejection. Can be used to capture reasons, feedback, or other context.
</ParamField>

<ParamField body="actionCode" type="string">
  Not used for reject actions (only for custom actions)
</ParamField>

## Response Schema

<ResponseField name="workflowInstanceId" type="uuid" required>
  ID of the workflow instance
</ResponseField>

<ResponseField name="stepInstanceId" type="uuid" required>
  ID of the step instance that was rejected
</ResponseField>

<ResponseField name="actionType" type="string" required>
  The action that was taken: `reject`
</ResponseField>

<ResponseField name="workflowStatus" type="string" required>
  Current status of the workflow after the rejection: `rejected`, `waiting`, or `running` (if rejection path continues)
</ResponseField>

<ResponseField name="nextStepCode" type="string">
  Code of the next step in the rejection path, or `null` if the workflow terminated
</ResponseField>

## Rejection Transition Logic

When a task is rejected, the workflow engine looks for a transition with `action_type = 'reject'` from the current step:

- **If a rejection transition exists**: The workflow follows that path to the specified next step (e.g., back to the requester for revision)
- **If no rejection transition exists**: The workflow terminates with status `rejected`

<Note>
If `remark_required_on_reject` is enabled for the step, you **must** provide a `remark` value. This ensures rejection reasons are properly documented.
</Note>

## Cancellation of Other Tasks

When a task is rejected, all other `open`, `queued`, or `claimed` tasks in the same step instance are automatically cancelled. This prevents parallel approvers from taking action after a rejection has occurred.

## Error Handling

| Status Code | Description |
|-------------|-------------|
| 200 | Task rejected successfully |
| 403 | User is not assigned to this task |
| 404 | Task not found |
| 409 | Task status is not `open` (already completed, cancelled, or queued) |
| 422 | Remark is required but not provided, or no rejection transition defined |
