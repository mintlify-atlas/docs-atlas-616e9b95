## Endpoint

<CodeGroup>
```http
POST /api/v1/human-tasks/{task_id}/approve
```
</CodeGroup>

Approves a human task assigned to the current user. This convenience endpoint sets the action type to `approve` automatically.

## Authentication

This endpoint requires authentication. Include a valid Bearer token in the Authorization header.

## Path Parameters

<ParamField path="task_id" type="uuid" required>
  The unique identifier of the task to approve
</ParamField>

## Request Body

<ParamField body="actionType" type="string">
  Not required - automatically set to `approve`
</ParamField>

<ParamField body="remark" type="string">
  Optional comment or reason for the approval. **Required** if the step definition has `remark_required_on_approve` set to true.
</ParamField>

<ParamField body="payload" type="object" default="{}">
  Optional additional data to include with the action. Can be used to capture form data or additional context.
</ParamField>

<ParamField body="actionCode" type="string">
  Not used for approve actions (only for custom actions)
</ParamField>

## Response Schema

<ResponseField name="workflowInstanceId" type="uuid" required>
  ID of the workflow instance
</ResponseField>

<ResponseField name="stepInstanceId" type="uuid" required>
  ID of the step instance that was completed
</ResponseField>

<ResponseField name="actionType" type="string" required>
  The action that was taken: `approve`
</ResponseField>

<ResponseField name="workflowStatus" type="string" required>
  Current status of the workflow after the action: `running`, `waiting`, `completed`, `rejected`, `failed`, or `cancelled`
</ResponseField>

<ResponseField name="nextStepCode" type="string">
  Code of the next step in the workflow, or `null` if the workflow completed
</ResponseField>

## Approval Mode Behavior

The workflow's approval mode determines what happens after a task is approved:

### priority_chain
- Only the first task in the chain is `open`
- Others are `queued` and open sequentially
- Workflow advances when the active task is approved

### approve_any_one
- All tasks are `open` simultaneously
- Workflow advances when **any one** task is approved
- Other tasks are automatically cancelled

### approve_all
- All tasks are `open` simultaneously
- Workflow advances only when **all** tasks are approved
- If `nextStepCode` is `null` in the response, other approvals are still pending

### notify_all
- All tasks are `open` for visibility
- First approval advances the workflow

<Note>
When `remark_required_on_approve` is true for the step definition, the `remark` field is mandatory. The request will fail with a 422 error if the remark is missing.
</Note>

## Error Handling

| Status Code | Description |
|-------------|-------------|
| 200 | Task approved successfully |
| 403 | User is not assigned to this task |
| 404 | Task not found |
| 409 | Task status is not `open` (already completed, cancelled, or queued) |
| 422 | Remark is required but not provided |
