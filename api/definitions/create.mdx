## Overview

Create a new workflow definition in draft status. The workflow must include at least one start step, one terminal or end step, and valid transitions connecting all steps.

## Endpoint

```
POST /api/v1/workflow-definitions
```

## Authentication

This endpoint requires authentication. Include a valid authentication token in your request.

## Validation Rules

The workflow definition is validated with the following rules:

- **Step codes must be unique** within the workflow
- **Exactly one start step** is required (stepType: "start")
- **At least one terminal or end step** is required (isTerminal: true or stepType: "end")
- **All transition step codes must exist** in the steps array
- **Custom action transitions** must include an actionCode
- **No unconditional self-loops** are allowed
- **Only one unconditional transition** per source step and action type
- **Subworkflow steps** must include a subworkflowMapping
- **Non-terminal steps** must have at least one outgoing transition
- **All steps must be reachable** from the start step
- **Cycles without guards** are not allowed (cycles must have at least one step with maxVisitsPerInstance set)

## Response

Returns the created workflow definition with full details. Status code: 201 Created.

<ResponseField name="item" type="object" required>
  The created workflow definition (same structure as GET /api/v1/workflow-definitions/{definition_id})
</ResponseField>

## Example Request

```bash
curl -X POST https://api.example.com/api/v1/workflow-definitions \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "key": "simple-approval",
    "name": "Simple Approval Workflow",
    "description": "A simple two-step approval process",
    "steps": [
      {
        "stepCode": "start",
        "stepLabel": "Start",
        "stepType": "start",
        "assignmentPolicy": {
          "approvalMode": "priority_chain"
        },
        "associations": []
      },
      {
        "stepCode": "manager_approval",
        "stepLabel": "Manager Approval",
        "stepType": "human_task",
        "remarkRequiredOnReject": true,
        "assignmentPolicy": {
          "approvalMode": "approve_any_one"
        },
        "associations": [
          {
            "associationType": "role",
            "associationValue": "manager",
            "canApprove": true,
            "canReject": true
          }
        ]
      },
      {
        "stepCode": "end",
        "stepLabel": "End",
        "stepType": "end",
        "assignmentPolicy": {
          "approvalMode": "priority_chain"
        },
        "associations": []
      }
    ],
    "transitions": [
      {
        "fromStepCode": "start",
        "toStepCode": "manager_approval",
        "actionType": "approve",
        "transitionLabel": "Submit"
      },
      {
        "fromStepCode": "manager_approval",
        "toStepCode": "end",
        "actionType": "approve",
        "transitionLabel": "Approve"
      },
      {
        "fromStepCode": "manager_approval",
        "toStepCode": "end",
        "actionType": "reject",
        "transitionLabel": "Reject"
      }
    ]
  }'
```

## Error Responses

<ResponseField name="401" type="Unauthorized">
  Authentication token is missing or invalid
</ResponseField>

<ResponseField name="409" type="Conflict">
  Workflow key already exists
  
  ```json
  {
    "detail": "Workflow key already exists."
  }
  ```
</ResponseField>

<ResponseField name="422" type="Unprocessable Entity">
  Validation errors in the workflow definition
  
  ```json
  {
    "detail": {
      "errors": [
        "Exactly one start step is required.",
        "Step 'approval' must have at least one outgoing transition."
      ]
    }
  }
  ```
</ResponseField>

<Note>
New workflow definitions are created in "draft" status. Use the publish endpoint to activate them.
</Note>

<Note>
If builderLayout is not provided, the system will automatically generate a visual layout based on the steps and transitions.
</Note>
