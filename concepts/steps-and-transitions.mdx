## Overview

Workflows are directed graphs composed of **steps** (nodes) and **transitions** (edges). Steps define what happens at each point in the workflow, while transitions define how the workflow moves from one step to another based on actions.

<Info>
  Steps are **static definitions**, while step instances are **runtime executions**. A step can be visited multiple times in a single workflow instance.
</Info>

## Step Types

The engine supports six core step types:

### start

Every workflow must have exactly one start step. This is the entry point for workflow execution.

```json
{
  "stepCode": "start",
  "stepLabel": "Start",
  "stepType": "start",
  "isTerminal": false
}
```

<Note>
  Start steps are automatically entered when a workflow instance is created. They typically have a single `approve` transition to the first real step.
</Note>

### end

Terminal steps that complete the workflow. A workflow must have at least one end or terminal step.

```json
{
  "stepCode": "approved",
  "stepLabel": "Approved",
  "stepType": "end",
  "isTerminal": true
}
```

### decision

Logic branching points that evaluate conditions and route to different paths.

```json
{
  "stepCode": "check_amount",
  "stepLabel": "Check Invoice Amount",
  "stepType": "decision",
  "config": {
    "evaluationType": "expression"
  }
}
```

Decision steps use conditional transitions to determine the next step.

## Step Definition Schema

### workflow_step_definition

| Column | Type | Description |
|--------|------|-------------|
| `id` | uuid | Step definition ID |
| `workflow_version_id` | uuid | Parent version reference |
| `step_code` | text | Unique code within version |
| `step_label` | text | Display label for UI |
| `description` | text | Business description |
| `step_type` | text | One of: `start`, `end`, `human_task`, `system_task`, `subworkflow`, `decision` |
| `sequence_hint` | int | Ordering hint for builder UI |
| `allow_revert` | bool | Whether revert action is allowed |
| `remark_required_on_approve` | bool | Require comment for approve |
| `remark_required_on_reject` | bool | Require comment for reject |
| `remark_required_on_revert` | bool | Require comment for revert |
| `max_visits_per_instance` | int (nullable) | Loop guard - max times this step can be visited |
| `form_schema` | jsonb | Optional UI form configuration |
| `config` | jsonb | Step-specific configuration |
| `is_terminal` | bool | Mark as terminal (ends workflow) |
| `created_at` | timestamptz | Creation timestamp |

**Unique constraint:** `(workflow_version_id, step_code)`

## Transition Actions

Transitions define how workflows move between steps based on user or system actions.

### Action Types

#### approve

Positive progression - continues the workflow forward.

```json
{
  "fromStepCode": "manager_review",
  "toStepCode": "director_review",
  "actionType": "approve",
  "transitionLabel": "Approve",
  "description": "Manager approves, escalate to director"
}
```

#### reject

Negative outcome - typically ends workflow or routes to rejection path.

```json
{
  "fromStepCode": "manager_review",
  "toStepCode": "rejected",
  "actionType": "reject",
  "transitionLabel": "Reject",
  "description": "Manager rejects invoice"
}
```

#### revert

Sends workflow back to a previous step for corrections.

```json
{
  "fromStepCode": "director_review",
  "toStepCode": "manager_review",
  "actionType": "revert",
  "transitionLabel": "Send Back",
  "description": "Return to manager for clarification"
}
```

<Warning>
  Steps must have `allowRevert: true` for revert transitions to be available at runtime.
</Warning>

#### custom

User-defined actions with custom codes.

```json
{
  "fromStepCode": "review",
  "toStepCode": "escalation",
  "actionType": "custom",
  "actionCode": "escalate",
  "transitionLabel": "Escalate",
  "description": "Escalate to senior management"
}
```

### Validation Rules

From `workflows.py:114-120`:

```python
if transition.conditionExpression is None:
    signature = (transition.fromStepCode, transition.actionType)
    if signature in seen_unconditional:
        errors.append(
            "Only one unconditional transition is allowed per source step and action type."
        )
    seen_unconditional.add(signature)
```

**Rules:**
- Only **one unconditional transition** per `from_step + action_type` combination
- **Custom actions require `actionCode`**
- **Conditional transitions can coexist** with different priorities
- **Unconditional self-loops are forbidden**

## Conditional Transitions

Transitions can include conditional expressions evaluated against workflow context:

```json
[
  {
    "fromStepCode": "check_amount",
    "toStepCode": "auto_approve",
    "actionType": "approve",
    "conditionExpression": "$.amount < 1000",
    "priority": 1
  },
  {
    "fromStepCode": "check_amount",
    "toStepCode": "manager_review",
    "actionType": "approve",
    "conditionExpression": "$.amount >= 1000 && $.amount < 10000",
    "priority": 2
  },
  {
    "fromStepCode": "check_amount",
    "toStepCode": "director_review",
    "actionType": "approve",
    "conditionExpression": null,
    "priority": 3
  }
]
```

<Tip>
  Use `priority` to control evaluation order. Lower numbers are evaluated first. The first matching transition is taken.
</Tip>

## Step Instances (Runtime)

When a workflow enters a step, a **step instance** is created:

### step_instance

| Column | Type | Description |
|--------|------|-------------|
| `id` | uuid | Step instance ID |
| `workflow_instance_id` | uuid | Parent workflow instance |
| `step_definition_id` | uuid | Step definition reference |
| `attempt_no` | int | Retry attempt number |
| `visit_count` | int | Number of times this step has been visited in this workflow |
| `status` | text | `pending`, `active`, `waiting`, `completed`, `rejected`, `reverted`, `failed`, `cancelled` |
| `entered_at` | timestamptz | When step was entered |
| `started_at` | timestamptz | When step started executing |
| `completed_at` | timestamptz | When step completed |
| `waiting_since` | timestamptz | When step started waiting (for human tasks) |
| `actor_user_id` | uuid | User who completed the step |
| `result_action` | text | Action that completed the step |
| `result_payload` | jsonb | Additional result data |
| `error_payload` | jsonb | Error details if failed |

### Visit Counts

Steps can be revisited (e.g., after a revert). The visit count increments:

```python
# From runtime.py:268-277
def _get_next_visit_count(cursor, workflow_instance_id, step_definition_id) -> int:
    cursor.execute(
        """
        SELECT COUNT(*) AS count
        FROM step_instance
        WHERE workflow_instance_id = %s AND step_definition_id = %s
```

## Transition Resolution

At runtime, transitions are resolved based on action type:

```python
# From runtime.py:585-592
def _transition_for_action(cursor, workflow_version_id, from_step_definition_id, action_type, action_code):
    transitions = _get_transitions(cursor, workflow_version_id, from_step_definition_id, action_type)
    if action_type == "custom" and action_code:
        for transition in transitions:
            if transition["action_code"] == action_code:
                return transition
        return None
    return transitions[0] if transitions else None
```

Transitions are ordered by priority:

```python
# From runtime.py:253-265
def _get_transitions(cursor, workflow_version_id, from_step_definition_id, action_type: str):
    cursor.execute(
        """
        SELECT *
        FROM workflow_transition_definition
        WHERE workflow_version_id = %s
          AND from_step_definition_id = %s
          AND action_type = %s
        ORDER BY priority, created_at
        """,
        (workflow_version_id, from_step_definition_id, action_type),
    )
    return cursor.fetchall()
```

## Available Actions

The set of available actions for a step is determined by its outgoing transitions:

```python
# From runtime.py:280-293
def _available_actions(cursor, workflow_version_id, from_step_definition_id, allow_revert: bool):
    cursor.execute(
        """
        SELECT DISTINCT action_type
        FROM workflow_transition_definition
        WHERE workflow_version_id = %s AND from_step_definition_id = %s
        ORDER BY action_type
        """,
        (workflow_version_id, from_step_definition_id),
    )
    actions = [row["action_type"] for row in cursor.fetchall()]
    if not allow_revert:
        actions = [action for action in actions if action != "revert"]
    return actions
```

This list is stored in human tasks for display in the UI.
